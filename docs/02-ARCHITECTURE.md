## 2. 컴포넌트 분리 원칙

### 왜 Vercel + Fly.io 분리인가

| 작업 유형 | 처리 위치 | 이유 |
|---|---|---|
| 인증, CRUD, 짧은 API | Vercel Serverless | 빠른 콜드스타트, 자동 스케일, 비용 최적 |
| Playwright 헤드리스 브라우저 | Fly.io 컨테이너 | Vercel 60초 타임아웃 한계 + Chromium 패키지가 Vercel 람다 사이즈 초과 |

**Vercel Serverless의 한계**
- Hobby: 10초 / Pro: 60초 / Enterprise: 900초 함수 실행 시간
- Lambda 패키지 크기 50MB (Chromium 자체가 ~150MB)
- Cold start 부담

**해결**: 검색 요청이 들어오면 Vercel API가 큐에 잡만 등록하고 즉시 응답. 워커가 비동기로 처리하고 결과는 SSE(Server-Sent Events)로 스트리밍.

## 3. 검색 플로우 상세

```
[사용자] "5학년 사회 조선시대 PPT" 검색
   │
   ├─ 1. POST /api/search { query, filters }
   │     → searchId 발급, BullMQ에 잡 등록
   │     → 즉시 { searchId } 응답 (200ms)
   │
   ├─ 2. GET /api/search/stream?searchId=xxx (SSE 연결)
   │     → 클라이언트가 결과를 실시간 수신 대기
   │
   ▼
[Fly.io 워커]
   │
   ├─ 3. 큐에서 잡 수신
   │     → 사용자 자격증명 Firestore에서 조회
   │     → KMS로 복호화 (메모리 only)
   │
   ├─ 4. 5개 사이트 어댑터 Promise.allSettled 병렬 실행
   │     ├─ 각 어댑터: 세션 쿠키 재사용 또는 신규 로그인
   │     ├─ 검색 페이지 호출 → HTML 응답
   │     └─ Cheerio로 1차 파싱 → 결과 후보 N개 추출
   │
   ├─ 5. Gemini 2.0 Flash Lite 호출
   │     → HTML 후보를 공통 스키마로 구조화
   │     {
   │       title, summary, url, publishedAt,
   │       grade, subject, materialType, source
   │     }
   │
   ├─ 6. 사이트별 결과 도착 즉시
   │     → Redis Pub/Sub으로 SSE 채널에 publish
   │     → Vercel API가 사용자에게 스트리밍
   │
   └─ 7. 모든 사이트 완료 후
         → 통합·정렬·중복제거된 최종 결과 Firestore 캐시
         → 같은 사용자가 1시간 내 동일 검색 시 캐시 사용
```

## 4. 데이터 흐름 (자격증명 처리)

```
[등록 시점]
사용자 → 외부 계정 ID/PW 입력
  → Vercel API (HTTPS)
  → 즉시 Fly.io 워커로 자동 로그인 테스트
  → 성공 시:
    1. 새 DEK(Data Encryption Key) 생성 (사용자별)
    2. DEK로 외부 ID/PW 암호화
    3. KMS 마스터 키로 DEK 자체를 암호화
    4. 암호화된 DEK + 암호화된 자격증명을 Firestore 저장
  → DB 통째로 털려도 KMS 마스터 키 없으면 복호화 불가

[검색 시점]
워커가 자격증명 필요 → Firestore에서 암호화된 DEK + 자격증명 조회
  → KMS API로 DEK 복호화 (KMS 키는 IAM으로만 접근)
  → DEK로 자격증명 복호화 (메모리 only)
  → Playwright에 주입하여 로그인
  → 사용 후 메모리에서 즉시 폐기 (변수 null 처리)
```

## 5. 세션 관리 전략

같은 사용자가 5분 내 여러 검색 시 매번 로그인하면 사이트 봇 차단 위험. 따라서:

```
[첫 검색]
워커 → 5개 사이트 신규 로그인 → 쿠키 획득
       → Redis에 user:{uid}:session:{site} 저장 (TTL 30분)

[2회차 이후 검색]
워커 → Redis에서 세션 쿠키 조회
       → 쿠키만 주입하여 검색 (로그인 단계 스킵)
       → 401/리다이렉트 발생 시 → 신규 로그인 → 쿠키 갱신

[쿠키 만료]
TTL 만료 또는 검색 실패 시 자동 재로그인
```

## 6. 에러 처리 및 복구

### 사이트별 장애 격리
```typescript
const results = await Promise.allSettled([
  indischoolAdapter.search(query),
  iscreamAdapter.search(query),
  // ...
]);

// 한 사이트 실패해도 나머지는 정상 반환
// 실패한 사이트는 UI에 "일시적 장애" 표시
```

### 자동 복구 시나리오
| 장애 유형 | 감지 | 복구 |
|---|---|---|
| 쿠키 만료 | 401/302 응답 | 신규 로그인 후 재시도 1회 |
| 사이트 UI 변경 | 파싱 결과 0건 + 경고 | 알림 → 어댑터 패치 배포 |
| 캡차 출현 | 캡차 페이지 패턴 매칭 | 사용자에게 통지, 해당 사이트 일시 비활성화 |
| 사용자 비번 변경 | 로그인 실패 | 사용자에게 재등록 요청 푸시 |
| 네트워크 타임아웃 | 10초 초과 | 결과에서 해당 사이트 제외 + UI 표시 |

## 7. 모니터링 및 옵저버빌리티

### 로그
- Vercel: 자동 함수 로그 + Logflare 또는 Axiom 연동
- Fly.io: 자체 로그 + Grafana Cloud 연동
- 사용자 비밀번호·쿠키는 절대 로그에 남기지 않음

### 메트릭
- 사이트별 평균 응답 시간
- 사이트별 로그인 성공률
- 검색 P50/P95 응답 시간
- 일일 검색 건수, 사용자 수

### 알림
- 사이트별 로그인 성공률 < 90% 시 슬랙 알림
- Gemini API 에러율 > 5% 시 알림
- 워커 재시작 발생 시 알림
