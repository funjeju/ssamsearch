# CLAUDE.md

> **이 문서는 쌤서치 프로젝트의 헌법입니다.**
> 다른 모든 문서·코드·디자인 결정은 이 문서를 따릅니다.
> 개발 중 막히거나 의심이 들 때, 가장 먼저 이 문서로 돌아오세요.

## 0. 이 문서의 역할

쌤서치는 12개 이상의 문서로 구성됩니다. 각 문서는 자기 영역(보안, API, DB, 디자인)만 다루기 때문에, 문서끼리 충돌하거나 새로운 의사결정이 필요한 순간이 반드시 옵니다.

이 문서는 그때 펼치는 **상위 기준**입니다.

- AI 코딩 도구(Claude Code, Cursor 등)에게 컨텍스트를 줄 때 가장 먼저 읽혀야 할 문서
- 신규 개발자가 합류했을 때 가장 먼저 읽혀야 할 문서
- "이거 이렇게 해도 되나?"라는 질문이 생길 때마다 펼치는 문서

## 1. 서비스의 본질 (한 문장)

> **선생님이 자주 쓰는 5개 자료 사이트를 ID/PW 한 번 등록으로 묶어, 한 번의 검색으로 통합 결과를 받는 교사 전용 메타 검색.**

이 한 문장에 부합하지 않는 모든 기능 제안은 **기본값이 거절**입니다. 추가하려면 이 문장을 어떻게 강화하는지 명시적으로 설명할 수 있어야 합니다.

### 본질의 분해

| 키워드 | 의미 | 빠지면 안 되는 이유 |
|---|---|---|
| 선생님 자주 쓰는 | 교사 페르소나 정조준 | 일반 검색은 구글이 이미 하고 있음 |
| 5개 사이트 | 인디스쿨, 아이스크림, 티쳐빌, T셀파, 에듀넷 | "더 많이"가 아니라 "꼭 필요한 것만" |
| ID/PW 한 번 등록 | 자격증명 위탁 = 핵심 차별화 | 이게 가능하기에 통합이 가능 |
| 한 번의 검색 | 5개 따로 검색 → 1개 검색창 | 시간 1/6 가치의 원천 |
| 통합 결과 | 5개 결과를 하나의 리스트로 | 단순 모음이 아니라 비교·선택 가능한 형태 |
| 교사 전용 | 비전문가용 절대 아님 | 페르소나 흔들리면 PMF 실패 |
| 메타 검색 | 자료를 보유하지 않고 중개만 | 본문 저장 금지의 근거 |

## 2. 절대 원칙 (Non-negotiables)

이 항목들을 어기는 결정은 **다른 모든 문서·요구·일정을 무시하고 즉시 멈춥니다.** 우회·예외 없음.

### 2.1 본문은 저장하지 않는다

- 검색 결과의 메타데이터(제목, 요약, URL, 학년, 과목, 자료유형, 작성일)만 저장
- 자료의 본문, 첨부파일, 이미지, PDF는 절대 저장·캐시·전송하지 않음
- 사용자가 원본을 보려면 **반드시 원본 사이트로 이동** (새 탭)
- 이유: 저작권 + 원본 사이트 트래픽 환원 + 법적 안전판

**위반 예시 (절대 금지)**
- "사용자 경험 좋게 본문 미리보기 띄우자"
- "검색 빨리 하려고 본문 캐싱하자"
- "썸네일이라도 받아서 표시하자"

### 2.2 자격증명은 평문으로 어디에도 남지 않는다

- 외부 사이트 ID/PW: AES-256-GCM + KMS envelope encryption (필수)
- 메모리에 복호화된 비번이 존재하는 시간 = 로그인 1회 사용 직후까지
- 사용 후 즉시 `Buffer.fill(0)`으로 폐기
- **로그·에러 메시지·Sentry·콘솔 어디에도 비번이 들어가면 안 됨**
- DEK·세션 쿠키도 동일 원칙

**위반 시 신뢰 회복 불가능. 서비스 종료급 사고로 간주.**

### 2.3 사용자 동의 없는 자동 행동 금지

- 검색 외에 외부 사이트에서 어떤 행동도 자동 수행하지 않음
- 글 작성, 댓글, 좋아요, 다운로드, 결제 등 **상태를 변경하는 행위는 일체 금지**
- 사용자가 위임한 것은 "검색을 위한 로그인"뿐

### 2.4 사이트 운영진의 권리를 침해하지 않는다

- robots.txt 또는 명시적 약관에 따라 신호가 오면 즉시 해당 사이트 어댑터 비활성화
- 트래픽은 사이트당 분당 30회를 넘지 않도록 rate limit 강제
- 사이트 운영진이 합리적 협의를 요청하면 응한다

### 2.5 한국 개인정보보호법을 준수한다

- 개인정보 수집 동의 별도 분리
- 만 14세 미만 가입 차단
- 개인정보 유출 시 24시간 내 KISA 신고
- 사용자 계정 삭제 시 모든 자격증명 즉시 폐기 (소프트 삭제 금지)

### 2.6 검색이 느려지면 모든 것이 무너진다

- 검색 P50 응답 시간 4초 초과는 **버그로 취급**
- "정확도 위해 느려져도 됨"이라는 타협 불가
- 정확도-속도 충돌 시: 우선 빠르게 보여주고 백그라운드로 정확도 보완

## 3. 핵심 의사결정 기록 (ADR 요약)

미래의 본인 또는 새 개발자가 "이거 그냥 다른 방식으로 바꿀까?"라고 했을 때 돌아볼 근거.

### ADR-001: Vercel + Fly.io 이중 호스팅

- **결정**: 웹은 Vercel, Playwright 워커는 Fly.io
- **이유**: Vercel 함수 60초 타임아웃 + Chromium 패키지 50MB 제한 → 워커는 컨테이너 필요
- **변경 조건**: Vercel이 5분+ 함수 + 200MB 패키지를 합리적 가격에 지원하면 통합 검토

### ADR-002: Firestore 사용

- **결정**: PostgreSQL 아닌 Firebase Firestore
- **이유**: Next Curator와 인프라 일치, 서버리스 친화, 실시간 구독 활용 (검색 잡 상태)
- **변경 조건**: 복잡한 관계형 쿼리가 5개 이상 필요해지면 PostgreSQL 도입 검토

### ADR-003: Envelope Encryption (KMS + DEK)

- **결정**: 사용자별 DEK 생성 후 KMS 마스터 키로 DEK 암호화
- **이유**: DB 통째로 유출돼도 KMS 키 없이는 복호화 불가, 키 회전 용이
- **변경 조건**: **변경 불가.** 이 모델을 약화시키는 변경은 절대 금지

### ADR-004: Gemini 2.0 Flash Lite

- **결정**: HTML → 구조화 JSON 변환에 Gemini 사용
- **이유**: 비용 매우 저렴, JSON mode 안정적, Next Curator와 동일 스택
- **변경 조건**: 비용·정확도 문제 발생 시 OpenAI gpt-4o-mini로 fallback 가능 (어댑터화)

### ADR-005: SSE 스트리밍 (WebSocket 아님)

- **결정**: 검색 결과 스트리밍은 SSE
- **이유**: 단방향이면 충분, Vercel Edge Runtime 호환, 인프라 단순
- **변경 조건**: 양방향 통신 필요한 기능(채팅 등)이 추가되면 별도 WebSocket 채널

### ADR-006: shadcn/ui

- **결정**: 컴포넌트는 shadcn/ui 복사 방식
- **이유**: 디자인 자유도 + Tailwind 일관성 + 라이브러리 공급망 위험 회피
- **변경 조건**: 변경 불필요. 추가 컴포넌트도 동일 방식

### ADR-007: 모노레포 (pnpm workspace)

- **결정**: apps/web, apps/worker, packages/* 단일 저장소
- **이유**: 타입 공유, 어댑터 재사용, 일관된 배포 흐름
- **변경 조건**: 워커가 독립 팀 운영되면 분리 검토

## 4. 충돌 시 우선순위

문서·요구·결정이 충돌할 때 **위에서 아래로** 우선합니다.

```
1. 법적 의무 (개인정보보호법, 전자상거래법 등)
       ↓
2. 본 문서의 절대 원칙 (Section 2)
       ↓
3. 06-SECURITY.md (보안 설계)
       ↓
4. 01-PRD.md (제품 요구사항 핵심 가치)
       ↓
5. 02-ARCHITECTURE.md (시스템 아키텍처)
       ↓
6. 04-DATABASE.md / 05-API.md (구체 명세)
       ↓
7. 08-DESIGN_SYSTEM.md (UI·디자인)
       ↓
8. 11-ROADMAP.md (일정)
```

**예시 충돌과 판정**

| 상황 | 충돌 | 판정 |
|---|---|---|
| "디자인상 OTP 한 번만 받자" | 디자인 vs 보안 | **보안 승**: 외부 계정 등록·삭제·비번변경 시 OTP 필수 유지 |
| "검색 빠르게 본문 캐싱하자" | 성능 vs 절대 원칙 2.1 | **원칙 승**: 본문 캐싱 절대 금지 |
| "출시 일정 맞추려 KMS 생략하고 AES만" | 일정 vs 절대 원칙 2.2 | **원칙 승**: 출시 미루더라도 KMS 필수 |
| "유저 편의로 비번 7일 메모리 캐시" | UX vs 절대 원칙 2.2 | **원칙 승**: 사용 직후 폐기 |
| "사이트 봇 차단 회피 위해 분당 100회" | 기능 vs 절대 원칙 2.4 | **원칙 승**: 분당 30회 초과 금지 |

## 5. 의사결정 트리 (개발 중 막혔을 때)

### "이 기능 추가해야 하나?"

```
새 기능 제안
   │
   ├─ Section 1의 본질 한 문장에 직접 기여하는가?
   │     ├─ 아니오 → ❌ 거절. 다음 버전 백로그
   │     └─ 예 → 다음 단계
   │
   ├─ Section 2의 절대 원칙 중 어느 하나라도 약화시키는가?
   │     ├─ 예 → ❌ 즉시 거절. 우회 방법 없음
   │     └─ 아니오 → 다음 단계
   │
   ├─ MVP 단계에서 필요한가? (Phase 1)
   │     ├─ 아니오 → 📌 백로그 (Phase 3+)
   │     └─ 예 → 다음 단계
   │
   ├─ 5개 사이트 모두에서 동작하는가?
   │     ├─ 아니오 → ⚠️ 사이트 1개 의존 기능 → PoC 우선
   │     └─ 예 → ✅ 진행
```

### "이 코드 이렇게 짜도 되나?"

```
의심되는 코드
   │
   ├─ 비밀번호·자격증명·DEK가 등장하는가?
   │     ├─ 예 → 06-SECURITY.md 다시 읽기 → 의심되면 PR 리뷰 강제
   │     └─ 아니오 → 다음 단계
   │
   ├─ 외부 사이트의 상태를 변경하는가?
   │     ├─ 예 → ❌ 절대 원칙 2.3 위반. 코드 폐기
   │     └─ 아니오 → 다음 단계
   │
   ├─ 본문(자료 콘텐츠)을 메모리·DB·캐시·로그에 저장하는가?
   │     ├─ 예 → ❌ 절대 원칙 2.1 위반. 메타데이터만 추출
   │     └─ 아니오 → 다음 단계
   │
   └─ 4초 이내 사용자 응답 가능한가?
         ├─ 아니오 → 비동기 잡 큐로 분리
         └─ 예 → ✅ 진행
```

### "사이트 어댑터에 문제가 생겼다"

```
어댑터 실패
   │
   ├─ 사이트가 캡차를 띄움
   │     → 해당 어댑터 일시 비활성화 + 사용자 알림
   │     → 사이트 운영진 연락 검토 (절대 원칙 2.4)
   │
   ├─ 사이트 UI가 개편됨 (셀렉터 깨짐)
   │     → 어댑터 핫픽스 우선 배포 (12시간 이내)
   │     → 다른 4개는 정상 동작 유지
   │
   ├─ 사이트가 2FA 도입
   │     → 우리 OTP에 외부 OTP 입력 단계 추가 또는 어댑터 일시 보류
   │     → 절대 우회·자동화 금지 (절대 원칙 2.3)
   │
   └─ 사이트가 명시적 차단 요청
         → 24시간 내 어댑터 제거 (절대 원칙 2.4)
         → 사용자에게 투명하게 공지
```

## 6. 타협 가능한 것 vs 불가능한 것

### 타협 가능 (변경 자유)

- 디자인 (컬러, 레이아웃, 컴포넌트 스타일)
- 사이트 어댑터 추가·제거·교체
- AI 모델 (Gemini → OpenAI 등)
- 일정 (절대 원칙 충족 위해서라면 출시 미뤄도 됨)
- 결제 모델 (월/년/사용량)
- 마케팅 카피
- 도메인·서비스명

### 타협 불가능 (절대 변경 금지)

- 본문 저장 금지 정책
- 자격증명 KMS envelope encryption
- 원본 클릭 시 원본 사이트 이동 (트래픽 환원)
- 교사 페르소나 (일반인 대상 확장 금지)
- 메타 검색 모델 (자료 보유·재배포 금지)
- 사용자 동의 없는 외부 행동 금지

## 7. 새 개발자 / AI 도구 온보딩

### 새 개발자 첫날

1. 이 문서 (CLAUDE.md) 정독 — 30분
2. 01-PRD.md, 02-ARCHITECTURE.md — 1시간
3. 06-SECURITY.md — 1시간 (특히 자격증명 흐름)
4. 10-DEVELOPMENT.md 따라 로컬 환경 구축 — 2~4시간
5. PoC 인디스쿨 어댑터 코드 읽기 — 1시간
6. 첫 PR: 사소한 버그 수정 또는 문서 오타 → 리뷰 흐름 학습

### Claude Code / Cursor 등 AI 도구 사용 시

- 프로젝트 루트의 `CLAUDE.md`를 항상 컨텍스트에 포함
- 새 작업 시작 시 이 문서의 본질 한 문장(Section 1)을 명시적으로 인용
- AI가 절대 원칙(Section 2)에 어긋나는 코드를 제안하면 즉시 거부하고 원칙 재명시
- 의사결정 트리(Section 5)를 AI에게 따라가도록 요청

### AI에게 컨텍스트 줄 때 권장 프롬프트

```
나는 쌤서치(SsamSearch) 프로젝트를 개발 중이다.
프로젝트 루트의 CLAUDE.md 문서를 먼저 읽고, 그 안의
"본질 한 문장"과 "절대 원칙 6개"를 모든 답변에서 준수해라.

특히:
- 본문 저장 절대 금지
- 자격증명은 KMS envelope encryption 필수
- 외부 사이트에서 검색 외 행동 금지

이제 작업을 시작한다: [작업 내용]
```

## 8. 위반 발견 시 절차

### 코드 리뷰에서 위반 발견
1. PR 즉시 차단
2. 위반된 절대 원칙 번호 명시 (예: "Section 2.1 위반")
3. 대안 제시 또는 해당 기능 폐기

### 운영 중 위반 발견 (예: 비번이 로그에 노출)
1. 해당 기능 즉시 비활성화 (배포 롤백)
2. 영향 받은 사용자 식별 → 비번 변경 요청 알림
3. 24시간 내 영향 분석 보고서 작성
4. 개인정보 유출이면 KISA 24시간 내 신고
5. 동일 위반 재발 방지 메커니즘 구축 (자동화 우선)
